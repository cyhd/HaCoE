124 // NOTE: This driver is completely thread safe. With some very minor exceptions, you can call all these functions from
125 // multiple threads at the same time and the driver will keep everything straight. Because of this, you must call Add...(),
126 // Go...() and Get...() from the same thread for a particular set of requests. Internally the list of things to do and
127 // results are split by thread. This allows you to use multiple threads to make requests without accidently getting
128 // data from one thread into another. If you are Adding requests and then getting NO_DATA_AVAILABlE or similar error when
129 // Getting the results, then chances are you are running in different threads and don't realize this.
130 // SubNote: The driver tracks which thread a request is made in by the thread ID. If you kill a thread and then create a new one
131 // it is possible for the new thread to have the same ID. Its not really a problem if you call AddRequest first, but if you
132 // did GetFirstResult on a new thread you may actually get data from the thread that already ended.
133 // SubNote: As mentioned, the list of requests and results is kept on a thread by thread basis. Since the driver can't tell
134 // when a thread has ended, the results are kept in memory for that thread even though the thread is done (thus the reason for the
135 // above mentioned subnote). This is not a problem in general as the driver will clean it all up when its unloaded. When it can
136 // be a problem is in situations where you are creating and destorying threads continuously. This will result in the slow consumption
137 // of memory as requests on old threads are left behind. Since each request only uses 44 bytes and as mentioned the ID's will
138 // eventually get recycled, it won't be a huge memory loss, but it will be there. In general, even without this issue, we strongly
139 // recommend against creating and destroying a lot of threads. Its terribly slow and inefficient. Use thread pools and other
140 // techniques to keep new thread creation to a minimum. This is what is done internally,
141 // **************************************************
142 // NOTE: Continuing on the thread safety issue, the one big exception to the thread safety of this driver is in the use of
143 // the windows TerminateThread() function. As it warns in the MSDN documentation, using TerminateThread() will kill the thread
144 // without releasing any resources, and more importantly, releasing any synchronization objects. If you TerminateThread() on a
145 // thread that is currently in the middle of a call to this driver, you will more than likely leave a synchronization object open
146 // on the particular device and you will no longer be able to access the device from you application until you restart. On some
147 // devices, it can be worse. On devices that have interprocess synchonization, such as the U12, calling TerminateThread() may
148 // kill all access to the device through this driver no matter which process is using it and even if you restart your application.
149 // Just avoid using TerminateThread()! All device calls have a timeout, which defaults to 1 second, but is changeable. Make sure
150 // when you are waiting for the driver that you wait at least as long as the timeout for it to finish.
151 // **************************************************

342 // can pass in the desired channel #, then the number of data points desired in Value, and a pointer to an array to put the
343 // data into as X1. This array needs to be an array of doubles. Therefore, the array needs to be 8 * number of
344 // requested data points in byte length. What is returned depends on the StreamWaitMode. If None, this function will only return
345 // data available at the time of the call. You therefore must call GetResult() for this function to retrieve the actually number
346 // of points retreived. If Pump or Sleep, it will return only when the appropriate number of points have been read or no
347 // new points arrive within 100ms. Since there is this timeout, you still need to use GetResult() to determine if the timeout
348 // occured. If AllOrNone, you again need to check GetResult.

351 // returned, and therefore, the array must be 8 * number of scans requested * number of channels in each scan. Likewise

355 // and you request LJ_chALL_CHANNELS, you will get, Channel0, Channel1, Channel0, Channel1, etc. Once you have requested the
356 // data, any data returned is removed from the internal buffer, and the next request will give new data.

360 // and you request data from channel 0, then channel 1, then channel 0 again, the request for channel 0 the second time will

362 // the same until you've read the last channel in the scan, at which point your next block may be a different size.

377 /* example:
378 void StreamCallback(long ScansAvailable, double UserValue)
379 {
380  ... do stuff when a callback occurs (retrieve data for example)
381 }

384  tStreamCallback pCallback = StreamCallback;

388 // NOTE: the callback function is called from a secondary worker thread that has no message pump. Do not directly update any
389 // windows controls from your callback as there is no message pump to do so. On the same note, the driver stream buffer is locked
390 // during the callback function. This means that while your callback function is running, you cannot do a GetStreamData or any
391 // other stream function from a DIFFERENT thread. You can of course do getstreamdata from the callback function. You don't really
